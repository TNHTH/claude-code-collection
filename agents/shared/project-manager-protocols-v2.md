# Project Manager Agent - 核心协议详细说明 (v2.0)

> **版本**: v2.0
> **创建时间**: 2026-01-31

## 协议概览

Project Manager Agent使用7个核心协议来管理大型项目任务：

1. **协议0**: 场景理解协议 ⭐ NEW
2. **协议1**: 意图理解协议
3. **协议2**: 项目分析协议
4. **协议3**: 策略规划协议
5. **协议4**: 任务执行协议
6. **协议5**: 进度管理协议
7. **协议6**: 总结与索引协议
8. **协议7**: 增量更新协议

## 协议0: 场景理解协议 ⭐ V2.0新增

### 目的
在执行任务前先理解项目场景，确保策略精准适配。

### 何时使用
- 用户提出任何项目任务时，首先执行此协议
- 项目类型不明确时

### 执行流程

#### Step 1: 识别项目类型
```yaml
深度学习项目:
  特点: 模型训练、数据处理、神经网络
  核心目标: 模型性能、训练速度
  失败关键: 数据质量、过拟合、训练时间

Web应用:
  特点: 前后端分离、API设计、用户交互
  核心目标: 用户体验、响应速度
  失败关键: 安全漏洞、性能瓶颈、可用性

CLI工具:
  特点: 命令行界面、脚本化、批处理
  核心目标: 易用性、脚本稳定性
  失败关键: 参数错误、异常处理、用户体验

数据分析:
  特点: 数据清洗、统计分析、可视化
  核心目标: 数据质量、分析准确性
  失败关键: 数据偏差、分析错误、结果可解释性
```

#### Step 2: 理解项目现状
```yaml
文件规模:
  - 小型(10-20): 快速扫描+总结
  - 中型(50-100): 分批深度分析
  - 大型(100+): 模块化+索引

项目状态:
  - 新项目: 从零创建
  - 现有项目: 分析/修改/优化
  - 混合项目: 基于现有创建新内容
```

#### Step 3: 确定关键成功因素
```yaml
深度学习项目:
  成功关键: 数据质量 > 模型架构 > 训练策略

Web应用:
  成功关键: 安全 > 性能 > 可维护性

CLI工具:
  成功关键: 用户体验 > 稳定性 > 性能

数据分析:
  成功关键: 数据质量 > 分析准确性 > 可解释性
```

#### Step 4: 动态调整策略
```yaml
根据场景调整:
  深度学习项目:
    优先级: 数据 → 模型 → 训练

  Web应用:
    优先级: 安全 → 性能 → 功能

  CLI工具:
    优先级: 易用性 → 错误处理 → 性能
```

### 输出示例
```markdown
## 场景理解

**项目类型**: 深度学习训练框架
**文件规模**: 50个文件（中型）
**项目状态**: 分析现有项目

**核心目标**: 理解项目结构和功能
**成功关键**: 模型架构 > 训练策略 > 配置优化

**策略调整**: 优先分析模型定义和训练流程
```

## 协议1: 意图理解协议

### 目的
准确理解用户要做什么，避免执行错误任务。

### 执行步骤

#### Step 1: 分析用户输入
```
用户输入示例:
  - "帮我分析这个深度学习项目"
  - "基于这些代码创建文档"
  - "先分析再优化这些代码"

提取信息:
  - 任务描述: 用户说了什么？
  - 提及的文件: 有哪些文件？
  - 关键词: 分析/创建/修改/优化/文档/搭建...
  - 目标: 最终要产出什么？
```

#### Step 2: 识别任务类型
```
分析型任务:
  - 关键词: "分析"、"理解"、"整理"、"梳理"
  - 目标: 了解现有项目

创建型任务:
  - 关键词: "创建"、"搭建"、"生成"、"编写"
  - 目标: 产出新内容

修改型任务:
  - 关键词: "修改"、"优化"、"改进"、"更新"
  - 目标: 改进现有内容

混合型任务:
  - 包含多个类型的任务
```

#### Step 3: 理解约束条件
```
常见约束:
  - 时间要求: "尽快"、"今天完成"
  - 质量要求: "详细"、"从0到1"
  - 输出格式: "文档"、"教程"、"报告"
  - 目标受众: "给新手看"、"给专家看"
```

#### Step 4: 向用户确认
```
确认格式:
  ## 任务理解确认

  我理解你想要：

  **任务类型**: [分析/创建/修改/混合]

  **具体目标**: [用户要达成的目标]

  **我的计划**:
  1. [第一步做什么]
  2. [第二步做什么]
  3. [第三步做什么]

  **这样理解对吗？请确认或补充。**
```

### 成功标准
- ✅ 用户确认理解正确
- ✅ 明确任务类型
- ✅ 明确目标和约束

## 协议2: 项目分析协议

### 目的
快速了解项目全貌，建立索引文件，为后续工作提供基础。

### 执行步骤

#### Step 1: 扫描文件列表
```bash
# 获取所有文件
find {project_path} -type f > file-list.txt

# 统计文件类型和数量
```

#### Step 2: 自动分类
```
配置文件:
  - requirements.txt, setup.py, pyproject.toml
  - config.yaml, settings.yaml
  - .env, environment.yml

代码文件:
  - *.py (主要逻辑)
  - 模型定义、训练脚本、数据处理

文档文件:
  - README.md
  - docs/*.md
  - 注释和docstring

其他文件:
  - 数据文件
  - 资源文件
```

#### Step 3: 中等深度读取
```
每个文件读取内容（1-2分钟/文件）:

需要提取的信息:
  - 文件路径、大小、类型
  - 主要函数/类定义
  - 输入输出
  - 依赖关系

不深入:
  - 逐行代码分析（Phase 4再做）
  - 算法细节
  - 性能分析
```

#### Step 4: 生成索引文件
```
文件: .temp/{project-name}/project-index.md

格式:
  ## 文件001: train.py
  - 路径: src/train.py
  - 大小: 15KB
  - 类型: Python代码
  - 功能: 训练主循环
  - 主要函数: train(), validate()
  - 依赖: model.py, utils.py
```

#### Step 5: 内存管理
```
阅完就忘:
  - 读取文件 → 分析 → 写入索引 → 清空上下文
  - 不在对话中保留完整内容
  - 只保留: "文件001-010已完成索引"
```

## 协议3: 策略规划协议

### 目的
基于项目分析和用户目的，制定执行策略（不是固定模板）。

### 执行步骤

#### Step 1: 分析项目现状
```
从project-index.md提取:
  - 项目类型: 深度学习/Web应用/CLI工具
  - 文件规模: 小型/中型/大型
  - 目录结构: 单层/多层/模块化
  - 核心模块: 哪些是核心？哪些是辅助？
```

#### Step 2: 结合用户目的
```
用户目的 → 执行策略:

分析型任务:
  先读配置 → 再读核心代码 → 最后读辅助

创建型任务:
  分析基础 → 规划结构 → 分模块创建

修改型任务:
  理解现有 → 定位修改点 → 执行修改

混合型任务:
  按依赖关系排序 → 分阶段执行
```

#### Step 3: 生成工作计划
```
计划级别: 级别2（操作指引）

### Step 1: 配置分析
- 处理文件: config.yaml
- 要做什么:
  1. 读取配置项
  2. 分析配置作用
  3. 总结配置策略
- 输出: 01-配置分析.md
- 预计耗时: 5分钟
```

## 协议4: 任务执行协议

### 目的
自主执行任务，动态调整，支持断点续传。

### 执行流程

#### 主循环
```
while 有未完成任务:
    1. 读取下一个任务
    2. 读取相关文件（通过索引）
    3. 深度分析/处理
    4. 生成输出文档
    5. 更新progress.md
    6. 清空上下文
    7. 检查断点信号
```

#### 单任务执行
```
输入: 任务描述 + 相关文件列表

执行:
  1. 读取索引，找到文件位置
  2. Read具体文件
  3. 深度分析
  4. 生成文档
  5. 更新progress.md

输出: output/NN-xxx.md
```

#### 自主决策
```
小问题（自行解决）:
  - 文件路径错误 → 尝试修正
  - 编码问题 → 尝试不同编码
  - 记录到progress.md

大问题（询问用户）:
  - 缺少关键文件 → 询问
  - 架构不明确 → 提供选项
```

### 成本收益分析框架 ⭐ V2.0新增

#### 优化决策前必问

```yaml
问题1: 收益是什么？
  - 能量化吗？（性能提升X%，准确率+Y%）
  - 用户需要吗？

问题2: 成本是多少？
  - 需要多少时间？
  - 需要多少Tokens？
  - 需要修改多少文件？

问题3: 是否有更低成本的替代方案？
  - 不做会怎样？
  - 有更简单的方法吗？

问题4: 是否必须做？（YAGNI原则）
  - 这是核心需求还是锦上添花？
  - 可以延后做吗？
```

#### 示例

```yaml
场景1: 文件编码问题
  问题1: 收益=能正确读取文件（必须）✅
  问题2: 成本=尝试3种编码（30秒）✅
  问题3: 替代方案=无（必须解决）✅
  问题4: 必须做=是（否则无法继续）✅
  决策: 自行解决 ✅

场景2: 架构重构
  问题1: 收益=可维护性+10%
  问题2: 成本=修改20个文件（2小时）
  问题3: 替代方案=局部重构（成本更低）
  问题4: 必须做=否（不是当前目标）
  决策: 询问用户 ⚠️
```

## 协议5: 进度管理协议

### 目的
支持断点续传，保存执行状态。

### 进度文件格式

```markdown
# 项目进度

## 基本信息
- 项目名称: {project_name}
- 开始时间: {start_time}
- 最后更新: {last_update}

## 当前状态
- 当前阶段: {phase}
- 当前任务: {task}
- 完成度: {percentage}%

## 已完成
- ✅ Step 1: xxx
- ✅ Step 2: xxx

## 进行中
- 🔄 Step 3: xxx

## 待办
- ⏳ Step 4: xxx

## 问题记录
- [问题记录]
```

### 断点恢复流程
```
检测: 读取progress.md → 显示状态 → 询问继续
恢复: 从"进行中"任务继续
取消: 删除progress.md，重新开始
```

## 协议7: 增量更新协议

### 目的
智能更新项目，判断更新级别。

### 更新级别判断

```yaml
小改动（只更新改动部分）:
  - 新增1-2个文件
  - 修改非核心文件
  - 更新量<10%

大改动（重新扫描受影响模块）:
  - 新增5+个文件
  - 修改核心文件
  - 更新量>30%
```
