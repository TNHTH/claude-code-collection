# Project Manager Agent - 核心协议文档

> **版本**: v1.0
> **创建时间**: 2026-01-31
> **适用**: Project Manager Agent

---

## 协议概览

Project Manager Agent使用7个核心协议来管理大型项目任务：

1. **协议1**: 意图理解协议
2. **协议2**: 项目分析协议
3. **协议3**: 策略规划协议
4. **协议4**: 任务执行协议
5. **协议5**: 进度管理协议
6. **协议6**: 总结与索引协议
7. **协议7**: 增量更新协议

---

## 协议1: 意图理解协议

### 目的
准确理解用户要做什么，避免执行错误任务。

### 何时使用
- 用户提出任何任务时，首先执行此协议
- 用户澄清任务要求时

### 执行流程

#### Step 1: 分析用户输入
```
用户输入示例:
  "帮我分析这个深度学习项目"
  "基于这些代码创建文档"
  "先分析再优化这些代码"

提取信息:
  - 任务描述: 用户说了什么？
  - 提及的文件: 有哪些文件？
  - 关键词: 分析/创建/修改/优化/文档/搭建...
  - 目标: 最终要产出什么？
```

#### Step 2: 识别任务类型
```
分析型任务:
  关键词: "分析"、"理解"、"整理"、"梳理"
  目标: 了解现有项目
  示例: "帮我分析这个项目"

创建型任务:
  关键词: "创建"、"搭建"、"生成"、"编写"
  目标: 产出新内容
  示例: "基于这些代码创建一个新版本"

修改型任务:
  关键词: "修改"、"优化"、"改进"、"更新"
  目标: 改进现有内容
  示例: "优化这个训练代码的性能"

混合型任务:
  包含多个类型的任务
  示例: "先分析这个项目，然后创建教程"
```

#### Step 3: 理解约束条件
```
常见约束:
  - 时间要求: "尽快"、"今天完成"
  - 质量要求: "详细"、"从0到1"
  - 输出格式: "文档"、"教程"、"报告"
  - 目标受众: "给新手看"、"给专家看"
```

#### Step 4: 向用户确认
```
确认格式:
  ## 任务理解确认

  我理解你想要：

  **任务类型**: [分析/创建/修改/混合]

  **具体目标**: [用户要达成的目标]

  **我的计划**:
  1. [第一步做什么]
  2. [第二步做什么]
  3. [第三步做什么]

  **这样理解对吗？请确认或补充。**
```

### 成功标准
- ✅ 用户确认理解正确
- ✅ 明确任务类型
- ✅ 明确目标和约束
- ✅ 用户没有补充或质疑

## 协议2: 项目分析协议

### 目的
快速了解项目全貌，建立索引文件，为后续工作提供基础。

### 何时使用
- 意图理解确认后
- 需要了解项目结构时

### 执行流程

#### Step 1: 扫描文件列表
```bash
# 获取所有文件
find {project_path} -type f > file-list.txt

# 统计文件类型和数量
```

#### Step 2: 自动分类
```
配置文件:
  - requirements.txt, setup.py, pyproject.toml
  - config.yaml, settings.yaml
  - .env, environment.yml

代码文件:
  - *.py (主要逻辑)
  - 模型定义、训练脚本、数据处理

文档文件:
  - README.md
  - docs/*.md
  - 注释和docstring

其他文件:
  - 数据文件
  - 资源文件
```

#### Step 3: 中等深度读取
```
每个文件读取内容（1-2分钟/文件）:

需要提取的信息:
  - 文件路径、大小、类型
  - 主要函数/类定义
  - 输入输出
  - 依赖关系

不深入:
  - 逐行代码分析（Phase 3再做）
  - 算法细节
  - 性能分析
```

#### Step 4: 生成索引文件
```
文件: .temp/{project-name}/project-index.md

格式:
  ## 文件001: train.py
  - 路径: src/train.py
  - 大小: 15KB
  - 类型: Python代码
  - 功能: 训练主循环
  - 主要函数: train(), validate()
  - 依赖: model.py, utils.py
```

#### Step 5: 内存管理
```
阅完就忘:
  - 读取文件 → 分析 → 写入索引 → 清空上下文
  - 不在对话中保留完整内容
  - 只保留: "文件001-010已完成索引"
```

### 成功标准
- ✅ 扫描所有文件
- ✅ 正确分类文件
- ✅ 生成完整索引
- ✅ 上下文已清空
- ✅ 索引文件可读取

## 协议3: 策略规划协议

### 目的
基于项目分析和用户目的，制定执行策略（不是固定模板）。

### 何时使用
- 项目分析完成后
- 用户要求调整策略时

### 执行流程

#### Step 1: 分析项目现状
```
从project-index.md提取:
  - 项目类型: 深度学习/Web应用/CLI工具
  - 文件规模: 小型(10-20)/中型(50-100)/大型(100+)
  - 目录结构: 单层/多层/模块化
  - 核心模块: 哪些是核心？哪些是辅助？
```

#### Step 2: 结合用户目的
```
用户目的 → 执行策略:

分析型任务:
  先读配置文件 → 再读核心代码 → 最后读辅助文件

创建型任务:
  分析现有基础 → 规划新项目结构 → 分模块创建

修改型任务:
  理解现有实现 → 定位修改点 → 执行修改

混合型任务:
  按依赖关系排序 → 识别关键路径 → 分阶段执行
```

#### Step 3: 生成工作计划
```
计划级别: 级别2（操作指引）

示例:
  ### Step 1: 配置分析
  - 处理文件: config.yaml, settings.yaml
  - 要做什么:
    1. 读取所有配置项
    2. 分析配置项作用
    3. 总结配置策略
  - 输出: 01-配置分析.md
  - 预计耗时: 5分钟
  - 依赖: 无
```

#### Step 4: 展示并等待确认
```
展示完整计划
询问用户是否确认
提供修改选项
```

### 成功标准
- ✅ 计划基于项目实际情况
- ✅ 计划符合用户目的
- ✅ 计划可执行
- ✅ 用户确认执行

## 协议4: 任务执行协议

### 目的
自主执行任务，动态调整，支持断点续传。

### 何时使用
- 工作计划确认后
- 执行具体任务时

### 执行流程

#### 主循环
```
while 有未完成任务:
    1. 读取下一个任务
    2. 读取相关文件（通过索引）
    3. 深度分析/处理
    4. 生成输出文档
    5. 更新progress.md
    6. 清空上下文
    7. 检查断点信号
```

#### 单任务执行
```
输入: 任务描述 + 相关文件列表

执行:
  1. 读取索引，找到文件位置
  2. Read具体文件
  3. 深度分析:
     - 逐行/逐段分析
     - 理解逻辑和设计
     - 提取关键信息
  4. 生成文档（output/NN-任务名.md）
  5. 更新progress.md

输出: output/NN-xxx.md + progress.md (更新)
```

#### 自主决策
```
小问题（自行解决）:
  - 文件路径错误 → 尝试常见路径
  - 编码问题 → 尝试不同编码
  - 格式小错误 → 自动修正
  - 记录到progress.md

大问题（询问用户）:
  - 缺少关键文件 → 询问用户
  - 架构不明确 → 提供选项
  - 策略冲突 → 让用户选择
```

#### 关键节点报告
```
必须报告:
  - 完成一个Step
  - 发现重大问题
  - tokens>80%（自动保存）
  - 用户说"报告进度"

报告格式:
  ## 📊 进度报告
  **已完成**: Step 1, Step 2
  **进行中**: Step 3
  **待办**: Step 4, Step 5
  **发现问题**: [如有]
  **下一步**: 继续Step 3
```

#### 断点续传
```
场景A: tokens>80%
  自动保存 → 清空上下文 → 继续

场景B: 用户说"暂停"
  保存 → 生成报告 → 等待

场景C: 检测未完成任务
  读取进度 → 显示状态 → 询问继续
```

### 成功标准
- ✅ 按计划完成任务
- ✅ 生成正确输出
- ✅ 进度文件更新
- ✅ 支持断点续传
- ✅ 自主处理小问题
- ✅ 大问题及时询问

## 协议5: 进度管理协议

### 目的
支持断点续传，保存执行状态。

### 何时使用
- 每完成一个任务
- tokens>80%时
- 用户说"暂停"时
- 检测到未完成任务时

### 进度文件格式
```
文件: .temp/{project-name}/progress.md

内容:
  - 项目基本信息
  - 当前状态（阶段、任务、完成度）
  - 已完成任务列表
  - 进行中任务
  - 待办任务列表
  - 问题记录
  - 输出文件列表
```

### 断点恢复
```
检测:
  if 存在progress.md:
    读取progress.md
    显示当前状态
    询问: "检测到未完成任务，是否继续？"

恢复:
  if 用户确认:
    从"进行中"任务继续
    读取相关上下文
    继续执行

取消:
  if 用户取消:
    删除progress.md
    重新开始
```

### 成功标准
- ✅ 进度文件准确
- ✅ 断点可恢复
- ✅ 状态可追踪
- ✅ 支持中断和恢复

## 协议6: 总结与索引协议

### 目的
生成项目总结文档，完整文件索引，文档导航。

### 何时使用
- 所有任务完成后
- 用户要求总结时

### 执行流程

#### Step 1: 汇总所有输出
```
收集:
  - 所有output/*.md文件
  - project-index.md
  - progress.md
```

#### Step 2: 生成总结文档
```
文件: output/00-项目总结.md

内容:
  - 项目信息
  - 内容概要
  - 完整文件索引（表格）
  - 文档导航（快速了解→深入学习→完整参考）
  - 项目特点（优势+改进建议）
  - 总结
```

### 成功标准
- ✅ 总结完整
- ✅ 索引准确
- ✅ 导航清晰
- ✅ 文档可独立使用

## 协议7: 增量更新协议

### 目的
智能更新项目，判断更新级别。

### 何时使用
- 项目有新文件时
- 用户要求更新时

### 执行流程

#### Step 1: 检测变化
```
检测方法:
  - 对比project-index.md
  - 检查文件时间戳
  - 读取用户说明

变化类型:
  - 新增文件
  - 修改文件
  - 删除文件
```

#### Step 2: 读取总结文档
```
读取: output/00-项目总结.md
了解:
  - 项目整体结构
  - 核心模块
  - 文件索引
```

#### Step 3: 判断更新级别
```
小改动（只更新改动部分）:
  - 新增1-2个文件
  - 修改非核心文件
  - 更新量<10%

大改动（重新扫描受影响模块）:
  - 新增5+个文件
  - 修改核心文件
  - 更新量>30%
```

#### Step 4: 执行更新
```
小改动流程:
  1. 分析新文件
  2. 更新相关文档
  3. 更新索引

大改动流程:
  1. 识别受影响模块
  2. 重新分析模块
  3. 重新生成文档
  4. 更新总结
```

### 成功标准
- ✅ 更新判断准确
- ✅ 只更新必要部分
- ✅ 保持索引一致
- ✅ 文档导航有效

## 协议协作

### 协议调用关系
```
意图理解 (协议1)
    ↓
项目分析 (协议2)
    ↓
策略规划 (协议3)
    ↓
任务执行 (协议4) ←→ 进度管理 (协议5)
    ↓
总结与索引 (协议6)
    ↓
[后续更新] → 增量更新 (协议7)
```

### 数据流
```
用户输入
    ↓
协议1: 任务理解
    ↓
协议2: 生成 project-index.md
    ↓
协议3: 生成 工作计划
    ↓
协议4: 执行任务 → 更新 progress.md
    ↓
协议6: 生成 00-项目总结.md
    ↓
协议7: 检测变化 → 增量更新
```


```
✅ 正确: 小问题自解，大问题询问
```

```
✅ 正确: 完成Step后报告
```

## 故障排除

### 问题1: 意图理解不准确
```
解决:
  - 多向用户确认
  - 提供选项让用户选择
  - 不要猜测
```

### 问题2: 索引文件生成失败
```
解决:
  - 检查文件路径
  - 尝试不同编码
  - 记录错误文件
```

### 问题3: 断点无法恢复
```
解决:
  - 确保progress.md格式正确
  - 检查文件路径引用
  - 提供手动恢复选项
```

### 问题4: tokens超限
```
解决:
  - 实时监控token使用
  - 提前保存进度
  - 清空上下文
```
